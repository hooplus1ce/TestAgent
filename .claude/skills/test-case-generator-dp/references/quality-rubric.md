# 质量评分准则

> Phase 2 生成用例时，每条用例的「预期结果(L)」「级别(C)」必须通过本文件的判定标准。
> AI 自检时对照高级/中级清单逐条核对，禁止产出「系统正常运行」这类不可验证的废话。

## 一、预期结果可验证性（核心质量门）

预期结果必须满足：**能被人工或脚本客观判断真伪**。判断标准 = 「第三方读完这句，能否独立得出通过/不通过的结论」。

### ✅ 好的预期结果（可验证）

| 验证类型 | 示例 |
|---------|------|
| 数据范围 | 「所有制令单号均包含 `MO202606`，无不符合记录」 |
| 排序结果 | 「点击排序图标后，生产数量列从大到小排列，首行值为 `36719`」 |
| 状态保持 | 「弹窗关闭后，表格未刷新，行数仍为 15」 |
| 视觉变化 | 「弹窗消失，页面恢复至操作前的工单列表」 |
| 计数 | 「筛选后表格共 3 条记录，均为冲压车间」 |
| 校验拦截 | 「点击确定时弹出红色提示『请填写数量』，记录未提交」 |
| 字段联动 | 「选择物料后，单位字段自动填充为『个』且置灰不可编辑」 |

### ❌ 差的预期结果（不可验证，必须重写）

| 反例 | 问题 | 修改方向 |
|------|------|---------|
| 「系统正常响应」 | 无可观测信号 | 改为「200ms 内表格刷新，无报错弹窗」 |
| 「数据正确显示」 | 「正确」无定义 | 改为「制令单号列显示 MO202606270041，与查询条件一致」 |
| 「操作成功」 | 主观判断 | 改为「弹出绿色『操作成功』提示并自动消失，列表新增一行」 |
| 「功能正常」 | 空泛 | 改为具体的输入→输出对照 |
| 「页面跳转正确」 | 跳到哪？ | 改为「iframe URL 变为 /createMaterialOrder，标题显示『备料单创建』」 |

## 二、级别决策树

为每条用例分配 C 列级别时，**从上到下匹配，命中即定级**：

```
┌─ 是否阻塞主流程（不通过则核心业务无法进行）？
│   └─ 是 → 高级
│
├─ 是否核心业务功能（增/删/改/提交/审批）？
│   └─ 是 → 中级
│
├─ 是否常见交互（查询/排序/筛选/分页/详情查看）？
│   └─ 是 → 低级
│
├─ 是否状态流转关键节点（如待排产→已排产）？
│   └─ 是 → 中级
│
└─ 其余（边角交互、禁用态、罕见组合、纯展示） → 低级
```

### 级别定义

| 级别 | 含义 | 配色 | 典型场景 |
|------|------|------|---------|
| **高级** | 阻塞项 | 红 | 登录、进入模块、核心数据加载失败 |
| **中级** | 核心 | 橙 | 新增/提交/审批、状态流转、必填校验 |
| **低级** | 一般 | 黄 | 查询、排序、筛选、分页、导出 |

### 反模式（常见误判）

- ❌ DFS 衍生的弹窗用例全标低级 → 应按弹窗内操作性质判断（提交类中级，查看类低级）
- ❌ 排序用例标中级 → 排序是常见交互，应低级
- ❌ 必填校验标低级 → 校验拦截影响提交，应中级

## 三、前置条件 vs 测试步骤的边界（核心结构规则）

一条用例的完整链路：

```
前置条件（初始状态）→ 测试步骤1 → 测试步骤2 → ... → 测试步骤N → 预期结果（验证）
```

**每一步操作都必须在测试步骤中写明，不允许将中间状态偷渡到前置条件中。**

### 前置条件：只描述测试开始前的初始状态，不含任何操作

- ✅ 「1. 已登录系统  2. 在制造排产页面  3. 表格已加载  4. 未勾选行」
- ❌ 「4. 已进入排产详情」（这是操作后的状态，应放步骤）

### 测试步骤：包含从初始状态到验证前的所有操作

- ✅ 「1. 勾选一行待排产记录  2. 点击「批量排产」
     3. 等待切换到排产详情视图  4. 点击「返回」 5. 观察页面」
- ❌ 步骤中不写明怎么进入排产详情，直接跳到「进入排产详情」

### 判断口诀

- **前置条件 = 摄像机开机时画面里就有的东西**（状态、数据、页面位置）
- **测试步骤 = 摄像机开机后演员做的每一个动作**（点击、输入、滚动）
- 任何需要"做某个操作才能到达"的状态，都属于步骤，不是前置条件

> ⚠️ v1 旧规则「前置条件须含完整操作路径」「步骤禁止回溯父步骤」**已废弃**——它会把操作伪装成前置条件，破坏链路完整性。

## 四、高级自检清单（每条用例必过）

- [ ] 有可执行的具体步骤（J 列非空且编号连续）
- [ ] 预期结果可观测/可断言（对照上方「好的预期结果」）
- [ ] 用例编号全局唯一（A 列）
- [ ] 前置条件独立完整，无「同上」「同前」「见上文」等指代（I 列）
- [ ] 测试类型已标注（G 列）
- [ ] 用户已口头确认该用例
- [ ] 全批用例同时覆盖正向和负向场景
- [ ] 备注列保持为空（骨架用例在备注标注 `[待确认]` 除外）
- [ ] **预期结果经浏览器实际验证**：涉及 UI 反馈的预期结果（弹窗文案、按钮状态、提示消息）MUST 通过 `browser` 工具实际模拟操作并观察后写入。**禁止凭经验臆造**——未亲眼看到的反馈行为不得写入预期结果列。不确定时必须先执行交互验证
- [ ] **前置条件只描述初始状态**：写明测试开始前的页面位置、数据/筛选状态、勾选状态等，**不含任何操作**。「进入XX页」「已勾选行」这类需操作才到达的中间状态禁止写入前置条件，应放测试步骤（详见 §三）
- [ ] **测试步骤使用纯中文业务语言**：禁止在步骤中出现英文方法名（如 `getCellValue`、`mouse.click`、`colCount`）、英文字段名（如 `SLS`、`col14`）、技术框架名（如 `VTable`）。统一替换：VTable→「数据表格」/「数据报表」、col→「列」、row→「行」、scrollToCell→「滚动到该单元格」
- [ ] **用例正文使用业务语言**：禁止出现技术框架名（`VTable`、`props`、`fiber`），统一替换为业务领域用语

### 纯中文业务语言对照表

| ❌ 禁止（技术语言） | ✅ 应改为（业务语言） |
|-------------------|---------------------|
| 「调用 getCellValue 获取值」 | 「查看该单元格的值」 |
| 「点击 col14 单元格」 | 「点击商品识别码列的单元格」 |
| 「VTable 渲染表格」 | 「数据表格显示数据」 |
| 「执行 scrollToCell(0, 50)」 | 「滚动到第 51 行」 |
| 「fiber key 找到 vtableInstance」 | （不出现，属内部实现） |

## 五、中级自检清单

- [ ] 每条用例仅覆盖一个独立场景（一个验证点 D）
- [ ] 测试数据具体化（K 列值来自真实页面，非 xxx/测试值）
- [ ] 验证点有明确的可验证内容
- [ ] 同时覆盖正向和负向场景（同一功能至少 1 正 1 负）

## 六、用例去重规则（Phase 2 收尾执行）

DFS 探索会产生大量结构相似用例。导出前按以下规则合并：

**去重维度** = 相同的前置条件(I) + 相同的验证点(D) → 合并为一条，步骤用「或」连接。

| 合并前（2 条） | 合并后（1 条） |
|---------------|---------------|
| 查看弹窗内「物料编码」列 | 查看弹窗内明细数据（步骤：查看物料编码 / 物料名称 / 规格型号 列） |
| 查看弹窗内「物料名称」列 | 预期：各列均显示对应字段值，无空白错位 |

**不合并的情况**：验证点不同（如「查看数据」vs「验证数据联动」）、级别不同、前置条件不同。

## 七、覆盖完整性门禁

用例数量不足通常不是单条质量问题，而是覆盖建模不足。Phase 2 生成前必须先对照 `coverage-model.md` 完成覆盖矩阵。

### 导出前必须满足

- [ ] 每条正式用例都来自覆盖矩阵中的 `已验证` 场景
- [ ] 每个核心功能点至少覆盖 1 条正向路径和 1 条反向/异常路径
- [ ] 涉及输入的功能点至少覆盖 1 条边界/空值/格式类场景
- [ ] 涉及状态流转的功能点至少覆盖 1 条合法状态和 1 条非法状态或拦截场景
- [ ] 涉及弹窗的功能点至少覆盖打开、取消/关闭、确认或校验中的实际存在项
- [ ] 涉及表格的页面至少覆盖实际存在的 1 类表格交互，如选择、滚动、列宽、链接、排序、筛选、编辑
- [ ] 所有 `待验证`、`需用户确认`、`工具缺口` 均已在交付说明中列出

### 不允许的缩水行为

| 行为 | 问题 |
|---|---|
| 只生成页面加载、查询、导出 3 类浅用例 | 没有覆盖输入、状态、弹窗、表格交互和异常路径 |
| 用 1 条“筛选功能正常”覆盖所有筛选字段 | 不同字段的值域、操作符、空结果和组合查询风险不同 |
| 把多个按钮合并成“工具栏按钮可点击” | 不同按钮的前置条件、反馈和业务风险不同 |
| 只写成功路径，不写无选择/非法状态/必填缺失 | 缺少真实测试中的失败路径 |
| 因为未验证就直接不提候选场景 | 应列入 `待验证` 或 `需用户确认` 缺口 |
