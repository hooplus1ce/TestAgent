# 覆盖建模规范

本文件用于解决“页面分析后生成用例过少”的问题。核心原则：**先建覆盖模型，再生成用例**。覆盖模型属于测试设计脑层；真实点击、扫描、监听仍由 `drission-ui` MCP 执行。

## 一、强制阶段：Phase 1.5 覆盖建模

完成页面结构采集后，生成用例前，必须先输出覆盖模型。覆盖模型包括：

1. 页面资产清单：筛选字段、按钮、页签、表格列、行操作、弹窗、接口、状态字段、关键数据样本。
2. 可测功能点清单：每个资产对应哪些可验证行为。
3. 场景覆盖矩阵：每个功能点至少列出正向、反向、边界、状态、组合等候选场景。
4. 覆盖状态：标记为 `已验证`、`待验证`、`需用户确认`、`工具缺口`。
5. 缺口追踪：每轮探索结束后更新未覆盖项，不允许直接遗忘。

未完成覆盖模型时，禁止直接导出 Excel。

## 二、覆盖矩阵维度

| 维度 | 适用对象 | 应考虑的场景 |
|---|---|---|
| 正常路径 | 所有功能点 | 有效输入、合法状态、成功反馈、数据正确刷新 |
| 异常路径 | 输入、按钮、提交、接口 | 必填缺失、非法格式、重复提交、无权限、接口失败 |
| 边界值 | 文本、数字、日期、数量、金额 | 空值、最小值、最大值、超长、特殊字符、起止日期相等 |
| 等价类 | 文本、下拉、状态字段 | 有效值、无效值、历史数据值、当前页面已有值 |
| 组合条件 | 筛选区、搜索区、状态筛选 | 单字段、多字段 AND、清空重置、空结果 |
| 状态流转 | 审批、提交、作废、关闭、启用 | 操作前状态、操作后状态、非法状态拦截 |
| UI 交互 | 弹窗、抽屉、页签、表格 | 打开、关闭、取消、确认、刷新、返回、禁用态 |
| 数据一致性 | 表格、详情、导出、接口 | 列表与详情一致、筛选后导出一致、接口与 UI 一致 |
| 幂等与重复操作 | 保存、提交、审核、导出 | 连续点击、重复提交、防重提示、按钮 loading/禁用 |
| 权限与可见性 | 角色、组织、菜单、按钮 | 按钮是否可见/可点、无权限提示、数据范围限制 |
| 性能与容量 | 查询、加载、导出、大表格 | 首屏加载、查询响应、导出耗时、大数据滚动 |

## 三、最低覆盖目标

按功能点类型设置最低覆盖数量。达不到时必须在缺口清单中说明原因。

| 功能点类型 | 最低覆盖 |
|---|---|
| 筛选字段 | 每个主字段至少 1 条正向；核心字段额外覆盖空结果、重置、组合查询 |
| 文本/数字输入 | 正向 1 条、空值/必填 1 条、格式或边界 1 条 |
| 日期范围 | 正常范围、起止相同、清空重置、非法范围或无结果 |
| 下拉字段 | 真实选项正向、未选择/清空、无效值或不可自由输入约束 |
| 普通按钮 | 有前置条件成功路径、无前置条件/禁用态、取消/关闭路径 |
| 提交/保存/审核 | 成功路径、必填校验、重复点击或状态不允许 |
| 删除/作废 | 取消、确认、无选择/非法状态拦截 |
| 批量操作 | 无选择、单选、多选、部分非法状态 |
| 表格列 | 展示正确、横向滚动/固定列、可交互列点击反馈 |
| 弹窗 | 打开、取消/关闭、确认、字段校验、数据回填或状态变化 |
| 导入/导出/打印 | 正常触发、筛选条件继承、无数据或异常提示 |

## 四、覆盖状态定义

| 状态 | 含义 | 是否可生成正式用例 |
|---|---|---|
| `已验证` | 已通过真实浏览器操作、页面反馈、表格数据或接口证据确认 | 可以生成正式用例 |
| `待验证` | 根据页面资产和测试设计方法推导出的候选场景，尚未执行 | 不生成正式用例；列入下一轮探索 |
| `需用户确认` | 涉及业务规则、角色权限、数据状态或破坏性操作，需要用户确认 | 可生成骨架用例，备注 `[待确认]` |
| `工具缺口` | 当前 MCP 工具无法稳定获取或操作 | 不生成交互用例；记录工具能力缺口 |

## 五、覆盖矩阵输出格式

每次页面分析后，先输出类似清单：

| 区域 | 功能点 | 场景 | 类型 | 状态 | 证据/下一步 |
|---|---|---|---|---|---|
| 筛选区 | 制单时间 | 正常日期范围查询 | 功能测试 | 已验证 | 表格时间列均在范围内 |
| 筛选区 | 制单时间 | 起止日期相同 | 边界值测试 | 待验证 | 下一轮执行日期选择 |
| 工具栏 | 删除 | 未勾选行点击删除 | 异常测试 | 已验证 | 提示“请选择记录” |
| 工具栏 | 删除 | 已审批记录删除拦截 | 业务规则验证 | 需用户确认 | 需确认审批状态规则 |
| 表格 | 缴入仓库列 | 横向滚动后列值保持 | UI/交互测试 | 已验证 | 滚动后列值未错位 |

## 六、生成规则

1. 只把 `已验证` 场景写成正式测试用例。
2. `待验证` 场景必须进入下一轮探索清单，不直接写入 Excel。
3. `需用户确认` 场景如用户要求先占位，可写骨架用例，并在备注标注 `[待确认]`。
4. 同一功能点至少覆盖 1 正向 + 1 反向；涉及输入时补 1 条边界；涉及状态时补 1 条状态流转。
5. 页面级用例不能替代功能点用例；“页面加载正常”不能覆盖按钮、筛选、弹窗、表格交互。
6. 去重只合并验证点相同的用例，不得为了减少数量合并不同风险类型。

## 七、覆盖率自检

导出前必须回答：

- 是否每个可见主按钮至少有 1 条点击反馈用例？
- 是否每个核心筛选字段至少有 1 条正向筛选用例？
- 是否至少包含筛选重置、空结果或组合查询之一？
- 是否覆盖无选择/禁用态/非法状态等反向路径？
- 是否覆盖弹窗的打开、关闭/取消、确认或校验？
- 是否覆盖表格至少 1 类交互：选择、滚动、列宽、链接、排序、筛选、编辑中的实际存在项？
- 是否仍有 `待验证` 或 `需用户确认` 缺口？若有，是否向用户说明？

如果任一问题答案为“否”，不得声称“覆盖完整”，只能说“已覆盖已验证区域，并列出剩余缺口”。
