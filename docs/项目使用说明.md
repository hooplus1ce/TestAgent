# TestAgent 项目使用说明

## 1. 唯一 MCP 架构

项目只包含一个 MCP 实现：`mcp-service/src/drissionpage_mcp/`。其职责划分如下：

| 层级 | 目录 | 职责 |
|---|---|---|
| MCP 服务 | `mcp-service/` | 浏览器、会话、表格、浮层、网络、角色隔离、证据、执行和报告 |
| Claude Skill | `.claude/skills/test-case-generator-dp/` | 测试策略、覆盖模型、字段规范、Excel 导出 |
| Codex Skill | `.agents/skills/test-case-generator-dp/` | Codex 工作流适配 |
| 回归测试 | `tests/`、`mcp-service/tests/` | 直接验证新 Python 包与 MCP 契约 |

Codex、Claude 和 Trae 均通过根 uv workspace 直接运行 `drissionpage_mcp` 包，对外名称
固定为 `drissionpage-mcp`。不得新增第二套服务目录或 Skill 私有启动命令。

## 2. 环境与配置

根 workspace 统一管理唯一虚拟环境和依赖锁：

```bash
uv sync --all-packages --all-groups
```

浏览器配置位于 `mcp-service/configs/dp_configs.ini`，模板为同目录下的
`dp_configs.example.ini`。配置中的 `../dp_profile` 指向项目根目录浏览器数据，迁移
仓库时不依赖机器绝对路径。

主要环境变量：

| 变量 | 用途 |
|---|---|
| `DRISSIONPAGE_MCP_PROFILE` | 默认 `full`，向模型暴露完整工具；`enterprise` 可主动压缩上下文 |
| `DRISSIONPAGE_MCP_CAPS` | `all` 或 capability 列表 |
| `HL_REMOTE_PORT` | Chrome 调试端口，默认 9222 |
| `HL_SCM_URL` / `HL_SCM_BASE_URL` | SCM 页面与站点地址 |
| `HL_SCM_USERNAME` / `HL_SCM_USERPWD` | 默认账号登录 |
| `HL_SCM_ROLE_<ROLE>_USERNAME` / `USERPWD` | 多角色审批回归账号 |

凭据可由 Agent 的安全环境注入，也可放入被 gitignore 的本地
根目录 `.env`；进程环境优先。不得写入 `.mcp.json`、Skill、测试用例或任何被 Git
跟踪的配置文件。

## 3. Agent 启动适配

| 平台 | 配置文件 |
|---|---|
| Claude | `.mcp.json` |
| Codex | `.codex/config.toml` |
| Trae | `.trae/mcp.json` |

| 平台 | 服务入口 |
|---|---|
| Codex / Claude / Trae | `uv run --package drissionpage-mcp drissionpage-mcp` |

三者的项目配置均设置 `DRISSIONPAGE_MCP_PROFILE=full` 和
`DRISSIONPAGE_MCP_CAPS=all`，因此模型获得同一完整 capability 工具目录。

修改配置后重启 Agent 会话。分别用 `claude mcp list`、Codex `/mcp` 或
`codex mcp list` 检查 `drissionpage-mcp`。

## 4. 标准探索流程

1. `connect(port=9222, target_hint=...)`
2. `browser_tabs` 打开或选择待测页
3. `check_session`，必要时 `refresh_session`
4. `enter_module(..., expand_filter=True)`
5. `get_active_frame`
6. `flow_start` 与 `flow_capture_page_state`
7. `capture_page_model`、`scan_filter_fields`、`scan_table`
8. DOM 业务动作使用 `explore_action`，表格动作使用 `table_action`
9. `flow_stop` 与 `generate_test_cases_from_flow`
10. `run_test_cases`、`generate_test_report`、`compare_regression_report`

正式测试用例必须基于真实证据，包含 `automation_recipe` 和业务断言，并至少真实试运行一次。

## 5. 多角色审批回归

每个账号使用独立 BrowserContext，按业务顺序串行切换：

```text
role_session_start(requester)
role_session_start(dept_manager)
role_session_activate(requester) -> 创建并提交
role_session_activate(dept_manager) -> 查询待办并审批
role_session_activate(requester) -> 验证结果和权限
role_session_close(...) -> cleanup
```

除凭据外，还应维护账号所属部门、角色权限矩阵、审批模板和路由、可重复测试数据及每个节点的预期结果。

## 6. 输出与验证

正式用例 JSON 保存到 `test_cases/<LEVEL1_PINYIN>_<MODULE_PINYIN>/`，通过：

```bash
uv run python .claude/skills/test-case-generator-dp/scripts/generate_from_json.py test_cases/<目录>/*.json
```

运行全部回归：

```bash
uv run pytest -q
```

标准 MCP 客户端必须能够初始化服务、读取 resources 并枚举完整工具列表。
